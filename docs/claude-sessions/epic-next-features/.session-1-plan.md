## Key Design Decisions and Rationale

### 1. Backward Compatibility Strategy
**Decision**: Append new columns to `--bash` output rather than insert them
**Rationale**: Existing tooling that parses SESSION lines with `awk '{print $1, $2, $3}'` continues to work
**Implementation**: Change from 7 columns to 9 columns, adding model/cli at the end

### 2. Configuration Hierarchy  
**Decision**: CLI flags > config file > hardcoded defaults
**Rationale**: Follows standard UNIX convention, allows per-run overrides while supporting project-wide defaults
**Implementation**: Load `.epic-config.json` first, then apply CLI flag overrides

### 3. Graceful Degradation
**Decision**: All new features default to "off" state (timeout=0, retry=0, etc.)
**Rationale**: Ensures existing users experience no behavior change until they opt-in
**Implementation**: Zero/false defaults preserve current behavior exactly

### 4. Session Isolation for Model/CLI
**Decision**: Per-session frontmatter can override global settings
**Rationale**: Enables mixed-tool epics where different sessions use different models or CLIs
**Implementation**: Parse frontmatter keys, export via DAG plan, consume in run-sessions.sh

### 5. JSON Configuration Format
**Decision**: Use JSON for `.epic-config.json` rather than YAML or TOML
**Rationale**: Python stdlib JSON parsing, widely supported, simple structure
**Implementation**: Silent fallback on missing/malformed files prevents disruption

## Risks and Mitigations

### Risk: Backward Compatibility Break
**Scenario**: Changing `--bash` output format breaks downstream tooling
**Mitigation**: Append columns only, maintain existing column positions
**Verification**: Test with both old and new parsers

### Risk: Config File Parsing Errors
**Scenario**: Malformed JSON causes epic failures  
**Mitigation**: Wrap JSON parsing in try/catch, silent fallback to defaults
**Verification**: Test with missing, empty, and malformed config files

### Risk: Column Count Dependencies
**Scenario**: Session 03 expects new columns but Session 02 hasn't added them yet
**Mitigation**: Session 03 implements defensive parsing (7 or 9 columns)
**Verification**: Test run-sessions.sh with both old and new epic-dag.py output

### Risk: Documentation Drift
**Scenario**: Four documentation files become inconsistent
**Mitigation**: Quality gates check for presence of key terms in all files
**Verification**: Automated grep-based consistency checks
